# SQLite-базы и Reference Objects

SQLite в проекте используется не как вспомогательный кэш, а как полноценный промежуточный слой миграции. Через него проходят импорт, обработка, анализ, повторный экспорт и дозаполнение ссылок.

## Какие базы есть в проекте

По умолчанию рабочие файлы лежат в `BD/`.

Основные категории:

- `BD/<catalog>.db` - сырая БД после импорта из источника;
- `BD/<catalog>_processed.db` - обработанная БД для экспорта в приемник;
- `BD/reference_objects.db` - единая база ссылочных объектов;
- `BD/reference_objects_prod.db` - отдельная база ссылочных объектов для `--prod`;
- `BD/filters.db` или аналогичные файлы - пользовательские фильтры для выборочных загрузок.

Отдельно от рабочих БД используются базы метаданных:

- `CONF/upp_metadata.db`
- `CONF/uh_metadata.db`
- `CONF/type_mapping.db`

## Зачем нужны отдельные БД по каталогам

Такой подход упрощает:

- изоляцию сценариев по объектам;
- повторную обработку без повторного чтения из 1С;
- локальную отладку конкретного каталога;
- просмотр данных через веб-интерфейс;
- параллельную работу команды по нескольким объектам.

## Что делает db_manager

`tools/db_manager.py` отвечает за общий низкоуровневый слой SQLite:

- подготовку путей к файлам;
- создание базы при отсутствии;
- открытие подключения с `PRAGMA foreign_keys = ON`;
- преобразование ссылочных полей перед записью.

Для ссылок в таблицах обычно хранится JSON вида:

```json
{
  "presentation": "ООО Ромашка",
  "uuid": "....",
  "type": "Справочник.Контрагенты"
}
```

Для перечислений хранится строка вида:

```text
Перечисление.ИмяПеречисления.Значение
```

## Reference Objects

`reference_objects` это отдельный механизм для фиксации ссылочных объектов на этапе экспорта, когда writer создает или обрабатывает объекты в приемнике и сохраняет их состояние для последующего контроля.

За него отвечает `tools/reference_objects.py`.

Таблица `reference_objects` содержит:

- `ref_uuid`
- `ref_type`
- `ref_presentation`
- `source_data`
- `filled`
- `filled_at`
- `created_at`
- `parent_type`
- `parent_name`
- `parent_uuid`
- `field_name`

Ключевая идея:

- writer сохраняет ссылочный объект в `reference_objects`, когда он создает или обрабатывает его в контуре экспорта;
- если объект создан или зафиксирован неполностью, его можно зарегистрировать с `filled = 0`;
- когда объект выгружается полноценно, он помечается как `filled = 1`;
- после основного экспорта незаполненные объекты можно догрузить отдельным проходом.

## Как reference_objects используется в процессе

Типовой цикл:

1. writer на этапе экспорта сталкивается со ссылкой на объект или сам создает объект в приемнике;
2. объект сохраняется в `reference_objects`;
3. основной каталог экспортируется;
4. отдельный запуск `--fill-unfilled` ищет незаполненные записи;
5. `tools/fill_unfilled_references.py` поднимает нужные processed БД и пытается выгрузить отсутствующие объекты.

Это особенно полезно для сложных графов зависимостей, где порядок экспорта не всегда можно идеально линейно выстроить.

## Что делает fill_unfilled_references

Скрипт:

- читает статистику по `reference_objects`;
- определяет подходящий `catalog_name` через `catalog_mapping.json` и `type_mapping.db`;
- ищет нужные UUID в обработанных БД;
- повторно использует существующих writer-ов для догрузки объектов;
- помечает найденные и выгруженные ссылки как заполненные.

## Как смотреть БД

Есть два рабочих способа:

- открыть файл любой SQLite-утилитой;
- использовать встроенный веб-просмотрщик таблиц через `web_interface.py`.

Веб-просмотрщик умеет:

- показывать список таблиц;
- читать строки таблицы;
- фильтровать и искать по содержимому.

## Что важно для сопровождения

- не смешивать сырые и обработанные БД в одной логике без явной необходимости;
- не удалять `reference_objects.db`, если после экспорта планируется `--fill-unfilled`;
- при работе с продом помнить, что используется отдельный `reference_objects_prod.db`;
- проверять, что имя таблицы в `*_processed.db` совпадает с тем, что ожидает writer.
