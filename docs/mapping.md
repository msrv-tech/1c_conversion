# Маппинг

Маппинг описывает, как объекты, поля, типы и значения перечислений из конфигурации источника переводятся в конфигурацию приемника. Это центральная часть проекта для сценария `УПП -> УХ`.

## Какие артефакты участвуют

- `CONF/type_mapping.db` - основная рабочая база маппинга.
- `CONF/type_mapping.json` - компактная выгрузка маппинга для чтения и ревью.
- `CONF/catalog_mapping.json` - реестр поддерживаемых каталогов и маршрутов `loader -> processor -> writer`.
- `CONF/chart_of_accounts_mapping.json` - отдельный маппинг плана счетов и субконто.
- `CONF/upp_metadata.db` и `CONF/uh_metadata.db` - источники структурных данных для генерации маппинга.

## Что лежит в type_mapping.db

По коду проекта используются как минимум такие сущности:

- `object_mapping` - соответствие объектов источника и приемника;
- `field_mapping` - соответствие полей, включая табличные части;
- `type_mapping` - соответствие типов полей;
- `enumeration_value_mapping` - соответствие значений перечислений.

Именно этот набор читает `tools/base_processor.py`.

## Автоматический маппинг

`tools/auto_mapping.py` строит черновой маппинг на основе метаданных обеих конфигураций.

Он использует:

- объекты и реквизиты из таблиц метаданных;
- совпадения по имени и `full_name`;
- разбор типов реквизитов;
- отдельную загрузку перечислений и их значений.

Автоматика полезна как стартовая точка, но не закрывает все предметные расхождения между `УПП` и `УХ`.

## Ручной маппинг

Для уточнения используются инструменты из `tools/manual_mapping.py`.

Они позволяют:

- вручную добавлять маппинги полей;
- помечать маппинг как ручной;
- хранить `search_method` для нестандартного поиска;
- заводить отдельную таблицу `enumeration_value_mapping`;
- выгружать результат в компактный JSON.

Ручной маппинг нужен в случаях, когда:

- имя поля изменилось;
- объект источника и приемника не совпадают один-в-один;
- нужно разрулить мультитипы;
- одно и то же перечисление в разных каталогах должно маппиться по-разному;
- writer ищет ссылку не по UUID, а по прикладному ключу.

## Маппинг типов

Типы используются минимум в трех местах:

1. при авто-сопоставлении полей;
2. при обработке ссылочных значений и перечислений;
3. при подготовке данных к writer.

После импорта `load_stage.py` может запускать `tools/update_reference_types.py`, который:

- берет `metadata_enumerations` из базы метаданных;
- строит индекс по значениям перечислений;
- заменяет временное JSON-представление на строку вида `Перечисление.<Имя>.<Значение>`.

Это важно, потому что дальше процессоры ожидают уже нормализованный тип перечисления.

## Маппинг значений перечислений

Значения перечислений поддерживаются не только на уровне типа, но и на уровне конкретных значений.

`MappingProcessor` в `tools/base_processor.py` делает следующее:

- определяет тип перечисления источника;
- ищет целевой тип перечисления;
- при наличии `enumeration_value_mapping` меняет конкретное значение;
- если точного маппинга нет, старается использовать типовой `target_type` поля или `type_mapping`.

Это покрывает сценарии, где:

- тип перечисления меняется полностью;
- тип тот же, но значения называются по-другому;
- один и тот же enum в разных каталогах требует разных правил.

## Ручные маппинги и search_method

Поле `search_method` в `field_mapping` используется как служебная подсказка для нестандартного поиска и преобразования.

Типовые случаи:

- поиск по коду вместо UUID;
- поиск по наименованию;
- составной поиск по нескольким реквизитам;
- особая логика для договоров, серий, характеристик и других сложных ссылок.

Если в проект добавляется новый нетривиальный writer или processor, сначала надо проверить, не проще ли выразить его через `search_method`, чем размазывать логику по нескольким модулям.

## catalog_mapping.json

`CONF/catalog_mapping.json` не заменяет `type_mapping.db`, а решает другую задачу: описывает рабочий маршрут объекта по стадиям.

Для каждой сущности там обычно фиксируются:

- имя объекта 1С;
- `catalog_name`;
- модуль `loader`;
- модуль `processor`;
- модуль `writer`;
- ожидаемый тип загрузки;
- комментарий по бизнес-ограничениям;
- при необходимости таблицы табличных частей.

Это основной реестр того, что реально поддерживается в проекте.

## chart_of_accounts_mapping.json

План счетов выделен в отдельный JSON, потому что он используется не как обычный field mapping.

Поддерживаются два варианта значений:

- простая строка: прямой маппинг кода счета;
- объект с `target` и правилами `subconto1/subconto2/subconto3`.

Этот файл читает `MappingProcessor` через `tools/chart_of_accounts_mapper.py`.

## Практический порядок работы с маппингом

1. Обновить метаданные обеих конфигураций.
2. Пересобрать или актуализировать `type_mapping.db`.
3. Проверить `type_mapping.json` как человекочитаемую витрину.
4. Добавить ручные маппинги для конфликтных полей и перечислений.
5. Прогнать импорт и обработку конкретного каталога.
6. Проверить результат в `*_processed.db`.
7. Только после этого идти в экспорт.
